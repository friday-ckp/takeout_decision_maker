# 产品简报：外卖点餐决策器

## Executive Summary

一个帮你在 30 秒内结束"今天吃什么"这个问题的个人 Web 工具。

每天中午点外卖时，面对大量选项容易陷入选择困难，在外卖平台上反复刷却迟迟做不了决定，浪费时间、影响效率。现有外卖平台的设计目标是**延长用户停留时间**，与用户"快速决策"的真实需求背道而驰。

本产品以"打开即决策"为核心设计原则，提供转盘和扫雷两种趣味决策模式，让用户 3 步以内完成决策，彻底告别选择困难。

## The Problem

每天中午点外卖时，用户面临以下痛点：

- **选择困难**：外卖平台提供数百家餐厅，无从下手
- **时间浪费**：反复刷列表、看菜单，30 分钟过去了还没决定
- **平台设计对立**：外卖 App 刻意制造信息过载来延长用户停留，而非帮助快速决策
- **重复纠结**：每天面对同样的餐厅，仍然陷入相同的纠结循环

## The Solution

一个本地部署的个人 Web 工具，核心体验：

- **打开即决策** — 首屏直达决策界面，无弹窗、无推荐流、无广告
- **转盘模式** — 一键旋转，停哪吃哪
- **扫雷模式** — 盲点格子，第一个"炸"到的就是今天要吃的
- **决策前准备** — 可选填心情/口味偏好过滤候选，可选排除近期吃过的
- **餐厅管理** — 维护个人常点餐厅库，支持收藏和拉黑

## What Makes This Different

- **反外卖平台设计逻辑**：我们帮用户快速结束决策，不是延长停留
- **工具型产品**：不做 AI 推荐、不学习习惯、用户全权控制
- **趣味决策机制**：把"选择"变成"游戏"，降低心理负担
- **零数据依赖**：无需接外卖平台 API，完全离线可用

## Who This Serves

**主要用户：** 个人使用，自己维护餐厅库，每天独自决策
**次要场景：** 和朋友一起纠结吃什么，通过分享链接多人参与决策（v1.1）

用户画像：有选择困难症的上班族，每天中午要快速解决午饭，手边已有一批常点的外卖餐厅。

## Success Criteria

- 打开页面到做出决策 ≤ 30 秒
- 零学习成本，第一次用就会用
- 长期使用数据不丢失（MySQL 持久化）
- UI 高设计质量，用起来有愉悦感

## Scope

**v1.0 MVP 范围内：**
- 转盘模式 + 扫雷模式（两种决策方式）
- 餐厅管理（添加/编辑/删除/收藏/拉黑）
- 历史记录 + 决策排除
- 可选心情/口味过滤
- 每日重玩次数限制
- 多人分享链接 + WebSocket 实时协作

**明确排除：**
- 对接外卖平台 API
- 用户账号/注册登录系统
- 移动端 App
- AI 推荐或习惯学习

## Technical Overview

| 层级 | 技术 |
|------|------|
| 前端 | 静态 HTML / CSS / JS |
| 后端 | Node.js |
| 数据库 | MySQL（用户提供连接，环境变量配置） |
| 多人实时 | WebSocket |

本地部署，数据库连接信息通过 `.env` 文件配置。

## Vision

成为个人饮食决策的"仪式感工具"——每天中午打开它，快速完成决策，积累自己的饮食数据，偶尔和朋友一起玩扫雷决定今天吃什么。

## 需求演进备注

> 记录从头脑风暴到 PRD 的关键变更，供后续决策溯源。

| 变更项 | 头脑风暴原始想法 | PRD v0.5 最终决策 | 变更原因 |
|--------|----------------|------------------|----------|
| 候选数量上限 | 智能过滤最多显示 5 个候选 | 转盘上限 16，扫雷上限 12 | 5 个候选在加入收藏 2 倍权重后格子过少，游戏体验差；16/12 是在动画可读性与候选丰富度间取得的平衡 |
| 数据存储 | 未明确 | MySQL（用户提供连接） | 用户明确要求数据持久化且自有 MySQL 实例，放弃 localStorage 方案 |
| 多人模式入口 | 实时汇总揭晓结果（方案模糊） | 分享链接 + WebSocket 等待室 | 选择 Option B（分享链接）而非扫码/账号体系，保持零注册门槛 |
