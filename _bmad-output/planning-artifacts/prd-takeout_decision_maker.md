# 产品需求文档（PRD）：外卖点餐决策器

**版本：** v0.5
**日期：** 2026-03-27
**作者：** Friday
**状态：** 草稿，经三轮对抗性 Review 修订，多人协作逻辑补全，待最终确认

---

## 一、产品概述

### 背景
每天中午面对外卖平台大量选项，用户容易陷入选择困难，浪费时间。本产品是一个本地部署的个人 Web 工具，帮助用户在 30 秒内完成"今天吃什么"的决策。

### 核心目标
- 打开即决策，首屏直达决策入口
- 从打开到出结果不超过 3 步
- 工具服从用户，不自作主张

### 技术栈
| 层级 | 技术 |
|------|------|
| 前端 | HTML / CSS / JS（静态） |
| 后端 | Node.js |
| 数据库 | MySQL（用户提供连接，环境变量配置） |
| 多人实时 | WebSocket |

## 三、功能需求

### 3.1 首页

**FR-01 决策入口**
- 首屏显示「开始决策」按钮，点击后进入模式选择页
- 页面右上角展示心情 icon，可选填当天心情

**验收标准：**
- [ ] 首页加载完成后，「开始决策」按钮可见且可点击
- [ ] 点击按钮后跳转至模式选择页，跳转时间 < 300ms

---

### 3.2 新用户引导（空状态）

**FR-00 首次使用空状态引导**
- 当餐厅列表为空时（首次启动或数据全部删除），首页「开始决策」按钮替换为引导状态
- 引导流程：
  1. 首页展示提示："你还没有添加任何餐厅，先加几个吧！" + 「去添加餐厅」按钮
  2. 跳转至餐厅管理页，展示快速添加表单（默认展开，不需要用户主动点击）
  3. 用户添加至少 2 个餐厅后，首页「开始决策」按钮自动恢复可用状态
- 引导只在餐厅数量 < 2 时触发，不影响已有数据的正常用户
- **与多人受邀流程的优先级**：用户通过分享链接进入时（URL 含 `/session/` 路径），直接跳转等待室，**不触发** FR-00 引导流程；FR-00 仅在用户主动从首页发起决策时触发

**验收标准：**
- [ ] 餐厅数量为 0 时，首页「开始决策」按钮不可点击，展示引导提示
- [ ] 点击「去添加餐厅」跳转至餐厅管理页且快速添加表单默认展开
- [ ] 餐厅数量达到 2 个后，首页恢复正常决策入口

---

### 3.3 模式选择

**FR-02 决策模式选择**
- 展示两个选项：「转盘模式」和「扫雷模式」
- 用户每次决策时手动选择，无默认值

**验收标准：**
- [ ] 两种模式均可点击进入
- [ ] 选择后进入对应的决策界面

---

### 3.4 转盘模式

**FR-03 转盘决策**
- 进入转盘页前询问：「是否要加入新选项？」
  - 是 → 展示快速添加表单
  - 否 → 直接填充转盘（从候选餐厅中选取，自动排除最近吃过的）
- **转盘候选数量上限：16 个**（含收藏餐厅的重复份额）
  - 若候选餐厅（过滤后）≤ 16 个，直接进入转盘
  - 若候选餐厅 > 16 个，先展示候选列表让用户勾选本次参与转盘的餐厅（默认勾选全部，用户可取消），确认后进入转盘；勾选界面以**餐厅为单位**展示（不展开收藏的重复份额），取消勾选某餐厅即该餐厅完全不参与（含其2倍权重份额）
- 转盘展示各餐厅名称，点击「开始」旋转
- 旋转结束后高亮显示结果餐厅
- 展示「就这家了！」和「再转一次」按钮
- 每日「再转」次数上限可在设置中配置（默认 3 次），达到上限后禁用按钮

**验收标准：**
- [ ] 转盘至少包含 2 个候选时才可旋转，否则提示用户添加餐厅
- [ ] 候选超过 16 个时，展示勾选界面，用户确认后进入转盘
- [ ] 旋转动画流畅，结果随机
- [ ] 每日重转次数达到上限时，按钮变为禁用状态并展示提示
- [ ] 点击「就这家了！」后记录本次决策到历史，跳转至结果页

---

### 3.5 扫雷模式

**FR-04 扫雷决策**
- 展示 N 个翻扣的格子（N = 当前候选餐厅数量，上限 12 个）
- 用户点击任意格子，第一个点开的格子即为今日决策结果
- 结果揭晓后展示餐厅名称，并提供「就这家了！」和「再来一次」按钮
- 每日「再来」次数上限与转盘共享，默认 3 次

**验收标准：**
- [ ] 格子数量等于当前候选餐厅数量（最少 2 个，最多 12 个）
- [ ] 点击格子后翻开动画展示，随机分配结果
- [ ] 点击「就这家了！」后记录历史，跳转至结果页

---

### 3.6 餐厅管理

**FR-05 添加餐厅**
- 字段：店名（必填）、品类（选填，如川菜/日料/快餐等）、备注（选填）
- 支持批量导入（JSON 格式）

**批量导入 JSON Schema：**

```json
[
  {
    "name": "餐厅名称",          // 必填，string，最长100字符
    "category": "川菜",          // 选填，string，最长50字符
    "tags": ["重口", "辣"],      // 选填，string[]，每项最长20字符
    "note": "备注信息"           // 选填，string
  }
]
```

**导入错误处理策略：**
- 文件格式非法 JSON → 整体拒绝，提示"文件格式错误，请检查 JSON 格式"
- 单条记录缺少 `name` → 跳过该条，继续导入其余，最终提示"X 条成功，Y 条跳过（缺少店名）"
- `name` 与已有餐厅重复 → 跳过该条，提示重复数量
- 单次导入上限 100 条，超出则截断并提示

**FR-06 编辑餐厅**
- 可修改店名、品类、口味标签（tags）、备注

**FR-07 删除餐厅**
- 软删除，进入回收站，7 天后自动清除
- 提供回收站恢复入口

**回收站7天自动清除实现方式（惰性触发）：**
- 无需定时任务，采用惰性检查：每次用户访问回收站页面时，后端检查 `deleted_at` 字段，将超过7天的记录执行物理删除
- 触发时机：GET `/restaurants/trash` 请求时在返回数据前执行清理
- 若用户从不访问回收站，数据永久保留不清除（可接受，不影响主流程）

**FR-08 收藏餐厅**
- 收藏标记，收藏的餐厅在转盘/扫雷中出现概率为普通餐厅的 2 倍
- 收藏列表可单独查看

**2 倍概率实现方式：**
- **转盘模式**：构建候选列表时，收藏餐厅在列表中出现 2 次（占据双份扇形），普通餐厅出现 1 次。转盘旋转后若结果指向同一餐厅的两个扇形之一，均视为该餐厅被选中
- **扫雷模式**：格子总数 = 普通餐厅数 × 1 + 收藏餐厅数 × 2（仍受总数上限 12 个格子约束）；收藏餐厅对应 2 个格子，点开任意一个格子均展示该餐厅
- 若收藏餐厅数量导致扫雷格子超过 12 个，收藏餐厅仍占 2 格，但普通餐厅按随机抽样方式补齐至 12 格上限

**FR-09 拉黑餐厅**
- 永久移除，不再出现在决策候选中
- 黑名单可查看，支持解除拉黑

**验收标准：**
- [ ] 添加餐厅后立即出现在候选列表
- [ ] 拉黑后下次决策不再出现
- [ ] 删除后进入回收站，7 天内可恢复

---

### 3.7 结果页

**FR-15 决策结果展示**
- 大字显示餐厅名称（主视觉焦点）
- 展示餐厅品类、备注（若有）
- 展示本次使用的决策模式（转盘 / 扫雷）
- 提供两个主操作按钮：
  - 「就这家了！」→ 确认决策，写入 decision_history，结束本次流程
  - 「换一个」→ 消耗一次重玩次数，返回上一个决策模式继续
- 若当日重玩次数已达上限，「换一个」按钮置灰并展示提示："今天的纠结次数已用完，就它了！"
- 顶部显示返回首页的导航入口

**单人模式行为：**
- 「就这家了！」→ 写入 decision_history，跳转至首页
- 「换一个」→ 扣减发起人当日 replay_count，返回决策模式

**多人模式行为：**
- 结果页仅发起人展示「就这家了！」和「换一个」按钮
- 其他参与者展示等待状态："等待发起人确认…"
- 发起人点「就这家了！」→ 写入发起人 decision_history + 更新 decision_sessions.status = done，广播会话结束，所有参与者跳转至会话结束页
- 发起人点「换一个」→ 扣减发起人 replay_count，触发新一轮多人决策（所有参与者同步返回决策界面）
- 次数耗尽时「换一个」置灰，仅发起人可见提示文案

**验收标准：**
- [ ] 单人模式：结果页正确展示餐厅名称、品类、备注
- [ ] 单人模式：点击「就这家了！」后写入历史并跳转首页
- [ ] 单人模式：点击「换一个」后扣减次数并返回决策模式
- [ ] 多人模式：非发起人看到等待状态，无操作按钮
- [ ] 多人模式：发起人确认后所有参与者同步跳转会话结束页
- [ ] 重玩次数耗尽时「换一个」置灰，显示提示文案

---

### 3.8 历史记录

**FR-10 决策历史**
- 每次点击「就这家了！」后记录：餐厅名称、决策时间、使用模式
- 历史记录可查看，展示最近 30 条

**FR-11 历史排除**
- 在转盘/扫雷决策时，可选择排除「最近 N 天内吃过的」（N 可配置，默认 3 天）

**验收标准：**
- [ ] 每次确认决策后历史记录自动保存
- [ ] 开启排除后，历史内的餐厅不出现在当次候选中

---

### 3.9 可选前置过滤

**FR-12 心情选择器**
- 首页右上角心情 icon，点击后展示表情选项（如 😊 😐 😴 😤）
- 选择心情后按默认映射规则过滤候选餐厅，用户可在设置中修改映射
- 每天可更改，跨日自动清空（以服务器本地日期为准）

**默认心情映射规则：**

心情处理分两种类型，实现方式不同：

**A. 过滤型**（缩小候选集）：

| 心情 | 说明 | 过滤逻辑 |
|------|------|----------|
| 😴 困倦 | 想吃清淡好消化的 | 保留 tags 包含"清淡"的餐厅；若结果 < 2 则降级为不过滤 |
| 😤 烦躁 | 想吃重口解压 | 保留 tags 包含"重口"的餐厅；若结果 < 2 则降级为不过滤 |

降级时在界面提示："当前心情没有足够匹配的餐厅，已显示全部候选"

**B. 排序型**（调整候选顺序，不缩小候选集）：

| 心情 | 说明 | 排序逻辑 |
|------|------|----------|
| 😊 开心 | 无限制，随便吃 | 不干预，按默认顺序展示全部候选 |
| 😐 一般 | 想吃熟悉的 | 有历史记录的餐厅排序靠前，其余餐厅仍可见（不排除） |

> 注：排序型逻辑不触发降级处理，候选集始终为全量。

**FR-13 口味偏好**
- 在模式选择页提供可选的口味标签：重口 / 清淡 / 咸 / 甜 / 随意
- 选择后过滤候选餐厅（需餐厅添加时打了对应标签）
- **降级逻辑**：若过滤后候选餐厅数量 < 2，自动降级为不过滤全量展示，并提示："没有足够匹配「{所选口味}」的餐厅，已显示全部候选"
- 选择「随意」等同于不过滤

**验收标准：**
- [ ] 心情/口味均为可选，不选不影响主流程
- [ ] 选择后候选餐厅数量实时更新

### 3.10 设置

**FR-14 全局设置**
- 每日重玩上限（默认 3 次）
- 历史排除天数（默认 3 天）
- 数据库连接信息（首次配置向导）

**多人场景重玩次数归属：**
- 「再来一次」操作由发起人触发，仅扣减**发起人**的当日 replay_count
- 受邀参与者不拥有独立的 replay_count，不受上限约束
- 次数耗尽时仅发起人的「换一个」置灰，决策流程就此结束

**跨日计数重置机制：**
- 以**服务器本地日期（YYYY-MM-DD）**为准，不使用 UTC
- 重置触发时机：用户发起决策请求时，后端检查 `daily_config.date` 是否等于今日日期
  - 若不等于（跨日）→ 在当次请求中自动创建新的 `daily_config` 记录，`replay_count` 归零
  - 若等于 → 使用已有记录的 `replay_count`
- 无需定时任务，惰性重置，按需触发
- 心情字段同样在跨日时清空（随 daily_config 新建而重置）

### 3.11 多人协作

**FR-16 生成分享链接**
- 在模式选择页提供「邀请朋友一起选」入口
- 点击后生成唯一分享链接（`/session/{share_token}`），有效期 24 小时
- 发起人可复制链接分享给朋友
- **候选餐厅归属规则**：多人会话使用**发起人**的餐厅列表，受邀者无需有自己的餐厅数据
- **候选快照机制**：会话创建时立即将发起人当前的候选餐厅 ID 列表（过滤后）序列化存入 `decision_sessions.candidate_snapshot`，后续决策以快照为准，不受发起人中途修改数据影响

**FR-17 加入决策会话**
- 受邀者通过链接进入，输入昵称后加入会话（无需注册）
- 系统为受邀者创建临时用户记录（users 表，name = 输入昵称）
- **匿名用户数据清理策略**：临时用户的 users 记录在其所参与的**最后一个会话过期满7天后**自动物理删除；清理触发时机为惰性检查（任意 API 请求时后台异步执行），session_participants 级联删除

**FR-18 实时等待室**
- 发起人看到等待室界面，实时显示已加入的参与者列表（WebSocket 推送）
- 发起人确认人员到齐后点击「开始决策」，触发所有参与者同步进入决策界面

**FR-19 多人同步决策**
- 所有参与者看到相同的转盘/扫雷界面（状态通过 WebSocket 同步）
- **转盘模式**：仅发起人可点击「开始」旋转，旋转动画和结果实时同步给所有参与者；其他参与者的「开始」按钮置灰
- **扫雷模式**：任意参与者均可点击格子；采用**服务端先到先得**策略——服务端收到第一个有效点击后立即锁定结果，后续点击忽略，并将揭晓结果广播给所有参与者；格子在结果广播前对所有人保持可点击状态（视觉乐趣）
- 结果确认后记录至 `decision_sessions.result_restaurant_id`
- **多人场景结果确认**：结果广播后展示结果页（含餐厅名、发起人姓名），由**发起人**点击「就这家了！」完成确认；确认后写入发起人的 `decision_history`，并将会话 status 更新为 `done`；其他参与者看到确认动画后自动跳转至会话结束页

**验收标准：**
- [ ] 分享链接生成后可在浏览器直接访问
- [ ] 受邀者加入后，发起人端实时更新参与者列表（延迟 < 1 秒）
- [ ] 决策结果在所有参与者端同步展示（延迟 < 500ms）
- [ ] 会话过期（24小时）后链接失效，访问时提示"会话已结束"

## 四、非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 首屏加载 < 2 秒（本地网络），单人决策结果响应 < 500ms |
| 实时同步 | WebSocket 消息广播延迟 < 500ms（局域网内）；单会话最大参与人数：10 人；断线后客户端自动重连，重连间隔最大 5 秒，重连成功后同步最新会话状态 |
| 可用性 | 极低学习成本；首次使用需完成数据库配置（一次性向导，约1分钟），配置完成后无需任何引导 |
| 数据安全 | MySQL 连接信息通过环境变量管理，不硬编码 |
| 兼容性 | 桌面端 Chrome / Firefox / Safari 最新版 |
| UI 品质 | 高设计质量，视觉精良（具体风格由 UX 阶段定义） |

## 五、数据模型

> 说明：所有表预留 `user_id` 以支持 v1.0 多人协作功能。v1.0 初始化时自动创建默认用户（id=1），单人使用时所有数据归属该用户，无需手动配置。

### users（用户）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| name | VARCHAR(50) | 用户名（分享链接场景显示用） |
| created_at | DATETIME | 创建时间 |

### restaurants（餐厅）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| user_id | INT FK → users.id | 所属用户 |
| name | VARCHAR(100) | 店名 |
| category | VARCHAR(50) | 品类 |
| tags | VARCHAR(200) | 口味标签，逗号分隔（已知技术债：不支持索引查询，候选数 < 500 条时性能可接受，v2.0 拆表优化） |
| note | TEXT | 备注 |
| deleted_at | DATETIME | 软删除时间，NULL 表示未删除 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后修改时间 |

### user_restaurant_relations（用户与餐厅关系）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| user_id | INT FK → users.id | 用户 |
| restaurant_id | INT FK → restaurants.id | 餐厅 |
| relation_type | ENUM('favorite', 'blacklisted') | 关系类型 |
| created_at | DATETIME | 创建时间 |

> 唯一约束：`UNIQUE(user_id, restaurant_id)`，每个用户对同一餐厅只能有一种关系（收藏或拉黑，二选一）。

### decision_history（决策历史）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| user_id | INT FK → users.id | 所属用户 |
| restaurant_id | INT FK → restaurants.id | 关联餐厅 |
| mode | ENUM('wheel', 'minesweeper') | 决策模式 |
| mood | VARCHAR(20) | 当次决策时的心情，可为 NULL |
| flavor_tags | VARCHAR(200) | 当次决策时的口味偏好，可为 NULL |
| decided_at | DATETIME | 决策时间 |

### daily_config（每日配置）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 自增主键 |
| user_id | INT FK → users.id | 所属用户 |
| date | DATE | 日期（服务器本地时区） |
| mood | VARCHAR(20) | 当天心情 |
| flavor_tags | VARCHAR(200) | 当天口味偏好 |
| replay_count | INT | 当日已重玩次数 |

> 唯一约束：`UNIQUE(user_id, date)`

### settings（全局设置）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 自增主键 |
| user_id | INT FK → users.id | 所属用户 |
| key | VARCHAR(50) | 配置键 |
| value | TEXT | 配置值（支持存储 JSON 格式的自定义映射规则） |

> 唯一约束：`UNIQUE(user_id, key)`

### decision_sessions（多人决策会话）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| host_user_id | INT FK → users.id | 发起人 |
| share_token | VARCHAR(64) | 分享链接 token，唯一 |
| mode | ENUM('wheel', 'minesweeper') | 决策模式 |
| status | ENUM('waiting', 'deciding', 'done', 'expired') | 会话状态 |
| candidate_snapshot | JSON | 会话创建时发起人候选餐厅 ID 列表快照，格式：`[1,3,5,...]` |
| result_restaurant_id | INT FK → restaurants.id | 最终决策结果，可为 NULL |
| created_at | DATETIME | 创建时间 |
| expires_at | DATETIME | 会话过期时间（默认24小时） |

### session_participants（会话参与者）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | 主键 |
| session_id | INT FK → decision_sessions.id | 所属会话 |
| user_id | INT FK → users.id | 参与用户（匿名用户也创建临时 user 记录） |
| joined_at | DATETIME | 加入时间 |
